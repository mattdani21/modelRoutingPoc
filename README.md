# Tessera Model Gate

Tessera Model Gate compares language models on approved business tasks.
It blocks a model when the model does not meet a task gate.
It stores the evidence for each decision.

## POC scope

The POC includes these functions:

- Load versioned benchmark tasks from YAML.
- Load model routes from YAML.
- call an OpenAI-compatible model endpoint.
- Run a safe demo without a model endpoint.
- Apply deterministic graders.
- Capture a human quality score from 1 to 5.
- Store results in SQLite, encrypted at rest when a key is configured.
- Export the evidence ledger as CSV.
- Block a route that is not approved for the task data class.
- Block a live run until the model has an approval flag, an artifact digest, and a counter-signed approval.
- Keep demo evidence separate from live evidence.
- Require an identified human reviewer before a quality decision.
- Enforce token-based access control with viewer, reviewer, and operator roles.
- Purge evidence past a configured retention window.
- Limit request rate and body size.
- Tag each run as champion or challenger and flag a regression against the champion.
- Repeat a run to expose variance.

The POC does not approve a model for production.
The POC does not send real client data.
The POC does not replace actuarial review.
The POC binds to loopback by default.
Do not bind it to a shared interface until access control and encryption are configured.
See `docs/PRE_DEPLOY_CHECKLIST.md` before any shared deployment.

## Start the service

Install a current stable Rust toolchain.
Then run these commands from this directory.

```bash
cargo test
cargo run
```

Open `http://127.0.0.1:8080`.
Select a task class.
Select a model set.
Select the execution mode. Demo is the default and uses synthetic responses.
Select the number of repetitions.
Select **Run benchmark**.

Running a task class runs every test case in the class across the selected routes.

Live mode calls a real endpoint. It requires an operator token, a model that is
approved for live, and a counter-signed approval. Set these up before you switch
the mode to live. See the control sections below.

You can also open `web/index.html` directly.
The page will use standalone demo data when the Rust service is not available.

## Offline demo

`web/demo.html` is a single self-contained file for a pitch or backup demo.
Open it directly in a browser. It needs no server, no model endpoint, and no
network. It uses synthetic data only and mirrors the dashboard and the gate
logic, including champion and challenger routing, human review, and the
evidence ledger export. A demo run can only reach the `demo_only` decision.

## Access control

The API is open when `config/access.yaml` is absent. This is only safe on
loopback for the local demo. For any shared deployment, copy the template and
set real tokens:

```bash
cp config/access.example.yaml config/access.yaml
```

Roles are `viewer` (read), `reviewer` (read and score), and `operator` (read,
score, and run). Generate a token with `openssl rand -hex 24`. Do not commit
`config/access.yaml`.

## Encryption, retention, and limits

Set a database key to encrypt the evidence ledger at rest with AES-256-GCM:

```bash
export MODEL_GATE_DB_KEY="$(openssl rand -base64 32)"
```

Set a retention window to purge old evidence:

```bash
cargo run -- --retention-days 30
```

`--rate-limit-per-minute` and `--max-body-bytes` cap request rate and body size.

## Champions and approvals

`config/champions.yaml` names one champion per task class. Every run is tagged
champion or challenger, and a challenger that fails a case the champion passed
is flagged as a regression and cannot be selected.

`config/models.yaml` `approved_for_live` is only a request. A live model also
needs a counter-signed record in `config/approvals.yaml` that matches the exact
artifact digest, or the service refuses to start:

```bash
cp config/approvals.example.yaml config/approvals.yaml
```

## Connect a model

Edit `config/models.yaml`.
Use an endpoint that supports `POST /v1/chat/completions`.
Set the exact model ID.
Set the exact quantisation.
Set the runtime and hardware.
Set the allowed data classes.
Set the token price when the route has a token price.
Replace the sample hosted gateway and prices before a real run.
Do not enter zero when a cost is unknown.
Use `null` until the cost basis is measured.

Do not put an API key in the YAML file.
Put the environment variable name in `api_key_env`.

## Add a benchmark

`config/benchmarks.yaml` is generated by `scripts/generate_benchmarks.mjs`.
It holds 15 synthetic test cases for each of the four task classes, each with a
deterministic oracle. Edit the generator and regenerate:

```bash
node scripts/generate_benchmarks.mjs
```

CI fails if the committed file drifts from the generator.

Give each test case a stable ID.
Use synthetic data during the POC.
Define a deterministic grader when one is possible.
Define a human rubric for tasks that need expert judgement.

Grow each pack toward 50 to 100 de-identified, approved cases before a
production decision. Create more cases for a high-risk task.

## Decision rule

Use this order:

1. Check the data route.
2. Check the deterministic gate.
3. Check the expert score.
4. Check the latency target.
5. Select the lowest total cost among the eligible models.

Do not rank a failed model above a passed model because it is cheaper.
Do not promote a result with the `demo_only` state.
